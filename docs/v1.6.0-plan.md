# v1.6.0 端口转发与网络管理 — 设计方案

---

## 一、需求概览

### 1.1 管理后台「网络」标签页

- 全局端口转发范围配置（默认 50000-60000）
- 管理员可修改，修改后自动应用到所有端口转发校验

### 1.2 VM / LXC 端口转发

在 VM 管理区 和 LXC 管理区的已分配卡片列表之外，各新增一个**「网络」标签页**，包含：

- 端口转发列表表格
- 添加端口转发
- 批量删除
- 自动加载已有转发规则 + 提取 VM/LXC 的 IP 地址同步进来

## 二、数据库设计

### 2.1 config 表新增配置项

| Key | 默认值 | 说明 |
|-----|--------|------|
| `forward:port_range_start` | `50000` | 端口转发范围起始 |
| `forward:port_range_end` | `60000` | 端口转发范围结束 |
| `forward:default_protocol` | `tcp` | 默认协议（tcp/udp/tcp+udp） |
| `forward:wan_interface` | `wan1` | 默认外网接口（管理员在「网络管理」tab 配置） |
| `forward:max_per_user` | `10` | 每用户最大转发规则数（包含 VM 和 LXC，0=不限制） |

### 2.2 新增 `port_forwards` 表

```sql
CREATE TABLE IF NOT EXISTS port_forwards (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    type TEXT NOT NULL,              -- 'vm' 或 'lxc'
    vm_id INTEGER,                  -- VM ID（type=vm 时）
    ct_id INTEGER,                  -- CT ID（type=lxc 时）
    name TEXT DEFAULT '',            -- 规则名称/备注
    ip TEXT NOT NULL,                -- 目标 IP（如 10.10.10.5）
    mac TEXT DEFAULT '',             -- MAC 地址（仅展示参考）
    internal_port INTEGER NOT NULL,  -- 内网端口
    external_port INTEGER NOT NULL,  -- 外网端口（范围由 forward:port_range 限制）
    protocol TEXT DEFAULT 'tcp',     -- tcp / udp / tcp+udp
    enabled INTEGER DEFAULT 1,       -- 1=启用 0=禁用
    source TEXT DEFAULT 'panel',     -- 'panel' 面板创建 / 'ikuai_sync' 启动同步导入
    sync_status TEXT DEFAULT 'synced', -- 'synced' 已同步 / 'orphan' 路由器上不存在 / 'pending' 待同步 / 'failed' 同步失败
    ikuai_id TEXT DEFAULT '',        -- ikuai 规则唯一标识（用于匹配去重）
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
)
```

## 三、后端 API 设计

### 3.1 网络配置

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/admin/network/config` | 获取端口转发配置（范围 + 默认外网接口） |
| `PUT` | `/api/admin/network/config` | 更新端口转发配置（端口范围 + 默认外网接口） |

### 3.2 端口转发 CRUD + 辅助接口

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| `GET` | `/api/port-forwards` | 获取端口转发列表；支持 `?type=vm` 或 `?type=lxc` 筛选 | 普通用户仅返回自己的规则，管理员返回全部 |
| `POST` | `/api/port-forwards` | 新增端口转发（校验外网端口是否在范围内 + 每用户数量上限） | 普通用户只能为自己的设备创建；管理员可为任意设备创建 |
| `PUT` | `/api/port-forwards/:id` | 编辑端口转发 | 仅规则所有者或管理员可编辑 |
| `DELETE` | `/api/port-forwards/:id` | 删除单条 | 仅规则所有者或管理员可删除 |
| `POST` | `/api/port-forwards/batch-delete` | 批量删除（传 id 数组） | 仅管理员 |
| `GET` | `/api/port-forwards/random-port` | 查询一个未被使用的随机端口 | 所有已登录用户 |
| `GET` | `/api/port-forwards/check-port` | 检查指定端口是否可用 | 所有已登录用户 |

### 3.3 IP 提取辅助  

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| `GET` | `/api/port-forwards/extract-ips` | 遍历所有分配的 VM/LXC，自动提取 IP 地址 | 所有已登录用户 |

### 3.4 ikuai 端口映射同步

端口转发的新增/编辑/删除通过 ikuai 内部 API（`Action/call dnat`）同步到路由器，详见第五章。

### 3.5 ikuai 接口列表查询

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/ikuai/interfaces` | 查询 ikuai 接口列表（名称、IP、状态、类型），用于前端外网接口选择下拉框 |

**说明**：VM 管理区的「网络」tab 打开弹窗时，设备下拉框**仅列出 VM**（不显示 LXC）；LXC 管理区同理仅显示 LXC。dashboard 用户面板同理仅显示当前用户所属的设备。

```
┌─ 添加端口转发 ──────────────────────────────────┐
│  📊 已使用 3 / 10 条                              │
│                                                   │
│  选择设备：  [▼ VM 201 - 10.10.10.5           ]   │
│              [  VM 202 - 10.10.10.7           ]   │
│              [  VM 203 (IP 未知 - 离线)       ]   │
│  目标 IP：     [10.10.10.5]（选中设备自动填入）    │
│  规则名称：    [自定义备注，如 "SSH 转发"]          │
│  协议：        [● TCP  ○ UDP  ○ TCP+UDP]          │
│  内网端口：    [22]                                │
│  外网端口：    [50022]  [🎲随机] [🔍检查]         │
│              └─ 可用范围: 50000-60000              │
│                                                   │
│           [取消]            [提交]                 │
└───────────────────────────────────────────────────┘
```

**编辑弹窗说明**：编辑时「选择设备」和「目标 IP」**可修改**，修改后视为重建规则：
- 后端先调用 `dnat / del` 删除旧规则 → 再调用 `dnat / add` 添加新规则
- 整个操作在一个事务中执行（本地标记 pending，全部成功后才标记 synced）
- 如果仅编辑端口、协议、名称，则调用 `dnat / edit` 更新

> 端口范围和外网接口选择由管理员在「网络管理」tab 统一配置，前端用户不可见。新增转发时后端自动使用系统配置的外网接口，并在提交时校验外网端口是否在配置范围内。

**外网端口辅助按钮**：

| 按钮 | 行为 |
|------|------|
| 🎲 随机 | 调用 `GET /api/port-forwards/random-port`，后端在 `port_range_start ~ port_range_end` 范围内查询已使用端口（本地 + ikuai），返回一个未被使用的随机端口，前端自动填入 |
| 🔍 检查 | 调用 `GET /api/port-forwards/check-port?port=50022`，后端检查该端口在本地和 ikuai 上是否已被使用。结果用颜色提示：✅ 绿色"端口可用" / ❌ 红色"端口已被占用，请更换" |

**冲突处理策略**：**阻断式**。外网端口冲突时提交按钮灰色禁用，端口输入框红色描边，右侧显示红色提示"端口已被占用"。用户必须先改端口或使用随机按钮选一个可用端口才能提交。

**校验规则**（前端 + 后端）：
- 外网端口 < port_range_start 或 > port_range_end → 红色提示，禁止提交
- 外网端口不可与已有规则冲突（同 IP 同端口）— 同时检查本地数据库和 ikuai 上已有规则
- 内网端口不可为空，1-65535
- **普通用户转发规则总数 >= `forward:max_per_user` 时，禁止新增**，前端提示"转发规则数量已达上限（N 条），如需新增请联系管理员"
- **管理员不受数量限制**

**同步规则**（后端自动）：
- 映射类型固定为"外网接口"，使用系统后台配置的默认映射类型
- 外网接口使用系统后台配置的默认接口（管理员在「网络管理」tab 设置）
- 新增/编辑/删除时通过 ikuai API 自动同步到路由器

### 4.5 数据提取逻辑

打开端口转发弹窗时，系统**自动检测**所有已分配 VM/LXC 的 IP 地址，渲染为设备选择下拉框：

- **LXC**：解析 `ct.config.net0` 中的 `ip=` 字段（格式如 `10.10.10.5/24`），提取 IP 地址
- **VM（QEMU）**：从 `vm.config.net0` 提取 MAC 地址，然后**通过 ikuai 软路由查询 DHCP 租约列表匹配 MAC → 返回对应 IP**

**ikuai DHCP 查询流程**（后端自动完成，用户无感）：
1. 后端登录 ikuai 管理界面（使用 `.env` 中配置的 `IKUAI_HOST` / `IKUAI_USER` / `IKUAI_PASSWORD`）
2. 调用 ikuai 内部 API 获取 DHCP 终端列表（MAC ↔ IP 对应关系）
3. 用 VM 的 MAC 地址匹配 DHCP 租约，提取 IP 地址
4. 返回前端渲染为下拉选项

**前端交互**：
- 下拉选项格式：`[名称] (VM/CT ID) - IP地址 [状态]`
  - 示例：`VM 201 (VM 201) - 10.10.10.5`
  - 示例：`CT 101 (CT 101) - 10.10.10.6`
- 选中设备后，目标 IP 输入框自动填入对应 IP
- 设备未知或 ikuai 查询失败时，显示提示"未能自动检测到设备，请手动输入目标 IP"

**兜底逻辑**：如果 ikuai 查询失败，IP 输入框仍可正常手动填写。

### 4.6 解绑/销毁级联清理

VM 或 LXC 被解除分配或销毁时，同步清理对应的端口转发规则：

| 触发操作 | 清理行为 |
|---------|---------|
| `DELETE /api/user/vms/:id`（解除分配 VM） | 删除 本地 `port_forwards` 中 `type='vm' AND vm_id={id}` 的所有记录，并逐个调用 ikuai `dnat / del` |
| `DELETE /api/user/lxc/:id`（解除分配 LXC） | 删除 本地 `port_forwards` 中 `type='lxc' AND ct_id={id}` 的所有记录，并逐个调用 ikuai `dnat / del` |
| `POST /api/lxc/:vmid/destroy`（销毁 LXC） | 同上，先清理转发规则，再执行 PVE 删除 |
| VM 到期自动清理（如果有） | 类似处理 |

**清理顺序**：
```
1. 查询该 VM/LXC 关联的所有端口转发规则
2. 对每条规则调用 ikuai dnat / del（忽略 404 错误—规则可能已不存在）
3. 全部处理后删除本地 port_forwards 记录
4. 如果 ikuai 不可达，本地记录标记 sync_status='orphan'，保留待下次启动同步清理
```

## 五、ikuai 端口转发自动同步

### 5.1 原理

面板新增/删除/编辑端口转发时，通过 ikuai 本地 REST API（端口 8080）直接在路由器上添加/删除端口映射规则（即 DNAT / 端口转发），不经过 PVE 宿主机。

### 5.2 对接方式（v3.6.x - v3.7.21）

ikuai 本地 API 为**纯 RESTful JSON API**，无需 cookie，使用 **MD5 签名认证**。

| 项目 | 说明 |
|------|------|
| 请求地址 | `http://{ikuai_IP}:8080/api`（端口固定 8080） |
| 请求方式 | POST |
| Content-Type | `application/json` |
| 认证方式 | MD5 签名 + Token |
| 前置要求 | 系统设置→基础设置→开启 API 接口 + 生成 API 密钥 + 添加白名单 IP |
| 频率限制 | 60 次/分钟 |

#### 签名算法

1. 将所有请求参数（不含 `sign`）按键名字母顺序排序
2. 按顺序拼接所有参数值，最后拼接 API 密钥
3. 对拼接后的字符串进行 MD5 加密，得到 32 位小写签名
4. 将 `sign` 字段添加到请求参数中

#### 登录获取 Token

```
POST http://10.10.10.1:8080/api
Body: {
  "method": "login",
  "username": "pve-panel",
  "password": "md5(明文密码)",  // 注意：是明文密码的 MD5
  "timestamp": 1717650000000,
  "sign": "32位MD5签名"
}
→ 返回 {"code":0, "data": {"token": "xxx", "expire": 3600}}
```

#### 功能调用（携带 Token + 签名）

```
POST http://10.10.10.1:8080/api
Body: {
  "method": "nat_portmapping_get",
  "token": "登录获取的Token",
  "timestamp": 1717650000000,
  "sign": "32位MD5签名"
}
→ 返回 {"code":0, "data": [...]}
```

### 5.3 方法映射

| 方法名 | 功能 | 备注 |
|--------|------|------|
| `nat_portmapping_get` | 查看所有端口映射规则 | 不加分页参数，返回全部 |
| `nat_portmapping_add` | 新增端口映射 | 参数见下方 |
| `nat_portmapping_set` | 修改端口映射 | 额外传 `id` |
| `nat_portmapping_del` | 删除端口映射 | 传 `id` |
| `monitor_lanip` → `dhcp_lease` `show` | 查询 DHCP 租约 + 在线设备 | 带分页参数 `TYPE,ORDER_BY,limit`，返回 `Data.data` 含 `mac,ip_addr,hostname,comment` |
| `network_interface` | 查询接口列表 | 返回 wan/lan 接口信息 |

### 5.4 端口映射参数对比

| 字段含义 | 旧 API（Action/call） | 新 API（local API） |
|---------|---------------------|--------------------|
| 目标 IP | `lan_ip` | `internal_ip` |
| 内网端口 | `lan_port` | `internal_port` |
| 外网端口 | `wan_port` | `external_port` |
| 协议 | `protocol` | `proto` |
| 接口 | `interface` | `interface` |
| 备注 | `comment` | `comment` |
| 启用 | `enabled: "yes"/"no"` | `enable: 1/0` |

**新增示例**：
```json
{
  "method": "nat_portmapping_add",
  "token": "...",
  "timestamp": 1717650000000,
  "proto": "tcp",
  "external_port": 50022,
  "internal_ip": "10.10.10.5",
  "internal_port": 22,
  "interface": "wan1",
  "comment": "SSH - VM 201",
  "enable": 1,
  "sign": "..."
}
```

**返回值格式**：
```json
{
  "code": 0,
  "msg": "success",
  "data": { "id": 2 }
}
```

### 5.5 同步时机

| 操作 | ikuai 同步行为 |
|------|---------------|
| 面板新增转发 | 1. 本地写入数据库（`sync_status: 'pending'`）→ 2. 调用 `dnat / add` → 3. 成功则标记 `synced`，失败则标记 `failed` 并提示用户 |
| 面板编辑转发（不改设备/IP） | 调用 `dnat / edit` 更新规则，同时更新本地数据库 |
| 面板编辑转发（改设备/IP） | 1. 本地标记 pending → 2. `dnat / del` 旧规则 → 3. `dnat / add` 新规则 → 4. 全部成功则更新本地标记 synced，任一失败则回滚本地变更并提示 |
| 面板删除转发 | 1. 调用 `dnat / del` → 2. 成功则删除本地记录，失败则提示"ikuai 不可达，请稍后重试"，保留本地记录 |
| 面板批量删除 | 逐个执行删除逻辑，孤立项（orphan）直接删除本地记录不调用 ikuai |
| **服务启动时** | **启动同步（startup sync）**：自动从 ikuai 全量拉取端口映射列表，与本地数据库做差异匹配后导入 |

**API 调用失败处理策略**：

| 失败场景 | 行为 |
|---------|------|
| 新增失败（ikuai 不可达） | 本地记录保留（`sync_status: 'failed'`），前端提示"同步失败，ikuai 不可达"；用户可点击重试 | 
| 新增失败（ikuai 返回拒绝） | 本地记录删除（ikuai 拒绝意味着此规则无效），前端提示失败原因 |
| 删除失败 | 保留本地记录，前端显示"路由器上可能存在残留规则"；用户可再次尝试删除，或启动同步时自动发现并标记 orphan |
| 编辑失败（改设备/IP，旧已删新未加） | 最严重的场景：本地恢复为旧值，前端提示"编辑失败，数据已恢复" |

**重试机制**：
- 对于标记为 `failed` 或 `pending` 的规则，在「网络」标签页提供「重试同步」按钮
- 手动点击后重新执行失败的 API 调用

### 5.5.2 ikuai Session 管理

ikuai 的 Web session 有默认过期时间（约 5-10 分钟无操作后自动失效）。

**Session 生命周期**：
```
[启动/首次调用] → 登录 ikuai，获取 cookie → 存入内存（全局变量）
    ↓
[每次 API 调用前] → 检查上次登录时间是否 > 4 分钟
    ↓ 是             ↓ 否
[重新登录]          [复用现有 cookie]
    ↓
[API 调用]
    ↓ 401 返回
[自动重试：重新登录 + 再次调用]
    ↓ 再次失败
[上报错误，停止操作]
```

**实现要点**：
- 全局维护 `lastLoginTime` 和 `cookieJar`（axios 实例的 cookie）
- 每次调用前检查时间间隔，4 分钟以上自动重登录
- 调用返回 401/403 时，自动重新登录后重试一次（最多 1 次，避免死循环）
- 登录失败时记录错误日志，不阻塞其他业务

**日志记录规范**（所有 ikuai API 操作均输出日志）：

| 事件 | 日志级别 | 输出内容 |
|------|---------|---------|
| 登录成功/失败 | `info`/`error` | `[ikuai] 登录成功 (host)` / `[ikuai] 登录失败: 原因` |
| API 调用成功 | `info` | `[ikuai] dnat/add 成功: 10.10.10.5:22→50022` |
| API 调用失败 | `error` | `[ikuai] dnat/del 失败 (401): 原因` |
| Session 过期重登 | `warn` | `[ikuai] session 过期，正在重新登录...` |
| 启动同步 | `info` | `[ikuai] 启动同步: 共 N 条规则，导入 M 条，跳过 K 条` |

### 5.5.3 启动同步 — 匹配规则与过滤逻辑

服务启动时执行 `syncFromIkuai()` 函数，流程如下：

```
1. 调用 ikuai API: dnat / show → 获取所有端口映射规则
2. 获取本地 port_forwards 表中所有记录
3. 对每一条 ikuai 规则，按以下优先级匹配本地记录：
```

**匹配规则**（第一条命中即视为已存在）：

| 优先级 | 匹配条件 | 示例 |
|--------|---------|------|
| ① 精确匹配 | `lan_ip` + `lan_port` + `wan_port` + `protocol` 全部一致 | `10.10.10.5:22 → 50022/tcp` |
| ② IP+端口匹配 | `lan_ip` + `wan_port` 一致（协议不同可更新） | 同上，忽略 protocol |
| ③ Comment 匹配 | `comment` 包含 `VM 201` 或 `CT 101` 等已分配 VM/LXC 标识 | `SSH 转发 - VM 201` |

**过滤规则**（仅导入满足以下条件的规则）：
- **目标 IP 属于已分配的 VM 或 LXC**（即 IP 存在于任一 VM/LXC 的 IP 列表中）
- 或 **comment 中包含已分配的 VM/LXC 名称或 ID**
- 不满足以上条件的规则视为"其他规则"，**跳过不导入**（避免导入无关的路由器端口映射）

**去重与更新策略**：

| 匹配结果 | 行为 |
|---------|------|
| ikuai 有 + 本地有，关键字段一致 | 跳过，不做变更 |
| ikuai 有 + 本地有，关键字段不一致 | 以 ikuai 为准更新本地（ikuai 是真实生效方） |
| ikuai 有 + 本地无，且符合过滤规则 | 自动导入本地，标记 `source: 'ikuai_sync'` |
| ikuai 有 + 本地无，不符合过滤规则 | 跳过（不导入无关规则） |
| ikuai 无 + 本地有 | 标记为 `sync_status: 'orphan'`（路由器上已不存在），前端灰色展示 + 提示"路由器上已不存在此规则"，保留手动删除 |

**Comment 标记规范**（面板创建时自动写入 comment）：

```
{规则名称} - {VM/CT ID}
```

示例：`SSH 转发 - VM 201`、`Web 服务 - CT 101`

这样同步回导入时就可以通过 comment 中的 VM/CT ID 快速匹配是否是面板管理的规则。

## 六、ikuai 配置说明

ikuai 的地址和账号密码通过 `.env` 文件配置，面板服务启动时自动读取，用于调用 ikuai 本地 REST API（DHCP 租约查询、端口映射同步、接口列表查询等）。**需要先在 ikuai 后台开启 API**：

1. 浏览器打开 ikuai 管理后台 → **系统设置 → 基础设置**
2. 下拉到最底部 → 勾选 **"开启API接口"**
3. 点击 **"生成 API 密钥"** → 复制 32 位密钥
4. 在 **"允许访问IP"** 中添加 PVE 面板服务器的内网 IP
5. 将密钥填入下方 `IKUAI_API_KEY`

```env
# ============ ikuai 软路由配置（v3.6+ 本地 API，端口 8080） ============
IKUAI_HOST=http://10.10.10.1
IKUAI_USER=pve-panel
IKUAI_PASSWORD=你的密码
IKUAI_API_KEY=爱快后台生成的32位密钥
# IKUAI_API_PORT=8080
```

> 以上信息仅后端使用，不在前端页面展示。
